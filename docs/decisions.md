# Принятые решения (ADR)

Документ описывает ключевые дизайн-решения, принятые при разработке OpenTelemetry SDK для OneScript.

## 1. Эталонная реализация - Java SDK

**Решение**: Моделировать API и внутреннюю архитектуру по образцу [opentelemetry-java](https://github.com/open-telemetry/opentelemetry-java).

**Причина**: Java SDK - одна из наиболее зрелых реализаций OpenTelemetry. Она полностью соответствует спецификации, хорошо документирована и использует паттерны (Builder, Provider, Processor), которые хорошо ложатся на OneScript.

**Следствия**:
- Builder pattern (ОтелПостроительSdk, ОтелПостроительРесурса и др.) вместо конструкторов с множеством параметров
- Централизованный SDK-объект вместо разрозненных провайдеров
- Автоконфигурация с теми же ключами (`otel.service.name`, `otel.exporter.otlp.endpoint` и др.)
- Та же иерархия семплеров (always_on, parentbased_always_on и т.д.)

## 2. Fluent API

**Решение**: Методы-сеттеры возвращают `ЭтотОбъект` для поддержки цепочек вызовов.

**Причина**: Удобство использования, привычный паттерн для Java-разработчиков, сокращение boilerplate-кода.

**Пример**:
```bsl
Запись = Логгер.СоздатьЗаписьЛога();
Запись.УстановитьТело("текст")
    .УстановитьСерьезность(ОтелСтепеньСерьезности.Info())
    .УстановитьАтрибут("key", "value");
```

## 3. Именование: русский язык

**Решение**: Все имена классов, методов, переменных и комментарии - на русском языке.

**Причина**: OneScript - русскоязычная среда разработки, целевая аудитория - русскоязычные разработчики 1С/OneScript. Русские имена повышают читаемость и снижают порог входа.

**Исключения**:
- `ОтелSdk` - аббревиатура, не переводится
- `ОтелBaggage` - термин OpenTelemetry, не имеет устоявшегося перевода
- `ОтелHttpТранспорт`, `ОтелGrpcТранспорт` - протоколы
- `ОтелW3CПропагатор` - стандарт W3C

## 4. Префикс `Отел` для всех публичных имен

**Решение**: Все классы и модули начинаются с префикса `Отел`.

**Причина**: В OneScript нет пространств имен. Префикс предотвращает конфликты имен при использовании библиотеки совместно с другими пакетами. `Отел` - краткий вариант от "OpenTelemetry".

## 5. Нет префикса `м` для переменных

**Решение**: Переменные именуются без префикса `м` - ни в классах, ни в модулях (например, `Имя`, `Атрибуты`, а не `мИмя`, `мАтрибуты`).

**Причина**: Префикс `м` не несёт семантической нагрузки и засоряет код.

## 6. Единый транспорт для всех экспортеров

**Решение**: Транспорт создается один раз и переиспользуется экспортерами спанов, логов и метрик.

**Причина**: Экономия ресурсов (одно HTTP/gRPC соединение), упрощение конфигурации. Каждый экспортер знает свой эндпоинт (`/v1/traces`, `/v1/logs`, `/v1/metrics`), но отправляет данные через общий транспорт.

## 7. gRPC: eager initialization в конструкторе

**Решение**: `ОтелGrpcТранспорт` открывает gRPC-соединение в конструкторе, а не лениво при первом вызове.

**Причина**: Ленивая инициализация вызывала race condition. Нативная компонента OPI_GRPC из библиотеки oint регистрируется глобально по ключу. Когда несколько фоновых потоков (BatchProcessor для спанов, BatchProcessor для логов, PeriodicReader для метрик) одновременно пытались открыть соединение, происходила повторная регистрация компоненты: `An item with the same key has already been added. Key: OPI_GRPC`.

Eager initialization в конструкторе гарантирует, что соединение открыто до запуска фоновых потоков.

## 8. Генерация ID из GUID

**Решение**: TraceId и SpanId генерируются из `УникальныйИдентификатор` (GUID), с удалением дефисов.

**Причина**: В OneScript нет криптографического генератора случайных чисел. GUID предоставляет достаточную уникальность для идентификаторов трассировки.

- TraceId (32 hex) = полный GUID без дефисов
- SpanId (16 hex) = первые 16 символов GUID без дефисов

## 9. Временные метки: наносекунды как строки

**Решение**: Временные метки представлены как строки с наносекундами от Unix epoch.

**Причина**: OTLP-протокол требует наносекунды. OneScript не поддерживает 64-битные целые числа, но строковое представление позволяет передавать значения в JSON без потери точности. Вычисление: `Цел(ТекущаяУниверсальнаяДата() - Дата(1970,1,1))` дает секунды, к которым добавляется `"000000000"` для наносекундной точности.

## 10. Автоконфигурация через configor

**Решение**: Автоконфигурация читает параметры через менеджер параметров configor, а не напрямую из переменных окружения.

**Причина**: configor поддерживает множество источников параметров (ENV, файлы, программное создание) с приоритетами. Это позволяет:
- Использовать стандартные переменные окружения (`OTEL_*`) в контейнерах
- Указывать параметры в файле конфигурации (`autumn-properties.json`) при локальной разработке
- Программно переопределять параметры в тестах
- Использовать один и тот же механизм конфигурации, что и в фреймворке autumn

## 11. NoOp-режим вместо отсутствия SDK

**Решение**: При `otel.enabled = false` создается полноценный SDK с NoOp-компонентами (семплер `ВсегдаВыключен`, без экспортеров).

**Причина**: Прикладной код не должен проверять, включена ли телеметрия. Все вызовы API (создание спанов, запись метрик) работают, но данные нигде не сохраняются. Это позволяет не менять код приложения при отключении телеметрии в продакшене.

## 12. Пакетные процессоры по умолчанию в автоконфигурации

**Решение**: Автоконфигурация всегда создает `ОтелПакетныйПроцессорСпанов` и `ОтелПакетныйПроцессорЛогов` (а не простые процессоры).

**Причина**: Простой процессор синхронно отправляет каждый спан/лог, что создает задержку в прикладном коде. Пакетный процессор накапливает данные и отправляет фоново, что критически важно для продакшена. Простой процессор остается для отладки и тестов.

## 13. Конвертер camelCase -> snake_case для gRPC; hand-written proto encoders для HTTP/protobuf

**Решение**: Внутренний формат данных SDK - camelCase (как в OTLP JSON). Для gRPC добавлен конвертер `ОтелКонвертерOtlpВProto` (camelCase -> snake_case + передача в OPI_GRPC). Для HTTP/protobuf реализованы отдельные ручные proto-кодировщики: `ОтелПротоКодировщикTrace`, `ОтелПротоКодировщикMetrics`, `ОтелПротоКодировщикLogs`.

**Причина**: OTLP JSON использует camelCase (`resourceSpans`, `traceId`), а protobuf - snake_case (`resource_spans`, `trace_id`). Вместо поддержки двух форматов внутри SDK, данные формируются в одном формате (camelCase) и конвертируются при необходимости.

Для HTTP/protobuf OPI_GRPC недоступен (он работает только с gRPC-транспортом), поэтому реализована прямая бинарная сериализация по wire-format protobuf согласно `.proto`-файлам OTLP. Кодировщики покрыты полевыми тестами (`ТестПротоКодировщик*Поля.os`) с декодированием через Python и официальные `.proto`-файлы.

## 14. Организация кода по подсистемам

**Решение**: Исходный код организован по подсистемам (Ядро, Трассировка, Логирование, Метрики, Экспорт, Пропагация, Конфигурация, Интеграции), а не по типам (Классы, Модули).

**Причина**: При 70+ файлах плоская структура становится неуправляемой. Организация по подсистемам:
- Отражает архитектуру OpenTelemetry
- Упрощает навигацию (все компоненты трассировки в одном месте)
- Позволяет легко добавлять новые подсистемы

Внутри каждой подсистемы - разделение на `Классы/` и `Модули/`. Регистрация файлов - через `lib.config`.

## 15. lib.config как источник истины для регистрации

**Решение**: Классы и модули регистрируются в `lib.config` (XML), а не в `packagedef`.

**Причина**: При сложной структуре подкаталогов `packagedef` не поддерживает автоматическое обнаружение файлов в поддиректориях. `lib.config` явно указывает пути к каждому файлу и работает при локальной разработке без установки пакета.

## 16. Только hyphen-minus (-), никогда em dash (-)

**Решение**: В коде и комментариях используется только символ дефис-минус (`-`, U+002D).

**Причина**: Em dash (`-`, U+2014) вызывает проблемы в OneScript/1C коде. Это символ Unicode, который может привести к ошибкам парсинга.
