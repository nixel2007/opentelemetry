# Архитектура OpenTelemetry SDK для OneScript

## Обзор

SDK реализует [OpenTelemetry SDK Specification](https://opentelemetry.io/docs/specs/otel/) и моделируется по образцу [Java SDK](https://github.com/open-telemetry/opentelemetry-java). Поддерживает три сигнала: трассировку, логирование и метрики. Экспорт данных осуществляется по протоколу OTLP через HTTP JSON или gRPC.

## Структура проекта

```
src/
  Ядро/              - Фундаментальные классы (SDK, Resource, Attributes, Context, Baggage)
  Трассировка/        - TracerProvider, Tracer, Span, SpanProcessors, Sampler
  Логирование/        - LoggerProvider, Logger, LogRecord, LogProcessors
  Метрики/            - MeterProvider, Meter, инструменты, агрегация, PeriodicReader
  Экспорт/            - Транспорты (HTTP, gRPC, InMemory), экспортеры, конвертер protobuf
  Пропагация/         - W3C TraceContext, W3C Baggage, композитный пропагатор
                        (B3 поставляется отдельным пакетом opentelemetry-propagator-b3)
  Конфигурация/       - Автоконфигурация из ENV/configor
  Интеграции/         - Аппендер для logos
tests/
  unit/               - Модульные тесты (~646), организованы зеркально к src/
  e2e/                - Интеграционные тесты с реальным OTLP-коллектором (testcontainers)
proto/                - .proto-файлы для gRPC-транспорта
```

## Ключевые компоненты

### ОтелSdk - централизованный объект

`ОтелSdk` - центральная точка доступа ко всем провайдерам и пропагаторам. Собирается через `ОтелПостроительSdk` (builder pattern). После сборки регистрируется глобально через `ОтелГлобальный`.

```
ОтелПостроительSdk
  .УстановитьПровайдерТрассировки(...)
  .УстановитьПровайдерЛогирования(...)
  .УстановитьПровайдерМетрик(...)
  .УстановитьПропагаторы(...)
  .ПостроитьИЗарегистрироватьГлобально()
```

Из любого места кода: `ОтелГлобальный.ПолучитьТрассировщик("имя")`, `ОтелГлобальный.ПолучитьЛоггер("имя")`, `ОтелГлобальный.ПолучитьМетр("имя")`, `ОтелГлобальный.ПолучитьПропагаторы()`.

### Ресурс (Resource)

`ОтелРесурс` описывает сущность, производящую телеметрию. Минимально необходимый атрибут - `service.name`. Ресурс создается один раз и передается во все провайдеры. Может строиться через `ОтелПостроительРесурса`, который автоматически добавляет системные атрибуты (telemetry.sdk.name, os.type, host.name, process.pid и др.).

### Атрибуты

`ОтелАтрибуты` - обертка над `Соответствие` с методами `Установить(Ключ, Значение)` и `Получить(Ключ)`. Используется повсеместно: в ресурсах, спанах, записях логов, точках данных метрик.

## Трассировка

### Пайплайн

```
TracerProvider -> Tracer -> Span -> SpanProcessor -> SpanExporter -> Transport
```

- `ОтелПровайдерТрассировки` - хранит ресурс, процессор и настройки семплирования. Создает `ОтелТрассировщик`.
- `ОтелТрассировщик` - создает спаны. Привязан к InstrumentationScope (имя + версия библиотеки).
- `ОтелСпан` - единица работы. Содержит имя, вид, атрибуты, события, линки, статус. При `Завершить()` передается процессору.
- `ОтелНоопСпан` - заглушка, когда семплер решил не записывать трассировку. Все методы - no-op.

### Контекст спана (SpanContext)

`ОтелКонтекстСпана` содержит:
- `TraceId` - 32 hex-символа (16 байт), генерируется из `УникальныйИдентификатор` (GUID) без дефисов
- `SpanId` - 16 hex-символов (8 байт), первые 16 символов GUID
- `TraceFlags` - флаг sampled (01) или not sampled (00)
- `TraceState` - vendor-specific key-value pairs
- `Удаленный` - флаг, что контекст пришел из другого сервиса (через пропагацию)

### Implicit Context Propagation

`ОтелКонтекст` - модуль, реализующий thread-safe стек текущих спанов. Каждый поток (ФоновоеЗадание) имеет свой стек. При вызове `Спан.СделатьТекущим()` (или `ОтелКонтекст.ПрисоединитьКонтекст(Контекст)`) контекст помещается на вершину стека и возвращается `ОтелТокенКонтекста`. Для восстановления вызывается `ОтелКонтекст.ОтсоединитьКонтекст(Токен)` либо `Токен.Закрыть()` (RAII). Токен и Спан - независимые сущности: `Спан.Завершить()` НЕ отсоединяет токен.

Это позволяет `НачатьСпан` автоматически находить родительский спан из контекста без явной передачи.

### Семплирование

`ОтелСемплер` поддерживает стратегии:
- `ВсегдаВключен` - записывать все
- `ВсегдаВыключен` - не записывать ничего
- `ПоДолеТрассировок` - записывать указанную долю (на основе TraceId)
- `НаОсновеРодителя` - наследовать решение от родительского спана, для корневых - применять вложенную стратегию

### Процессоры спанов

- `ОтелПростойПроцессорСпанов` - синхронный, для разработки. Сразу передает спан экспортеру.
- `ОтелПакетныйПроцессорСпанов` - асинхронный, для продакшена. Накапливает спаны в очереди, экспортирует пакетами по таймеру или при заполнении. Использует `ФоновоеЗадание` для фоновой отправки.
- `ОтелКомпозитныйПроцессорСпанов` - делегирует нескольким процессорам (fan-out).

### Лимиты спана

`ОтелЛимитыСпана` ограничивает:
- Максимальное количество атрибутов
- Максимальное количество событий
- Максимальное количество линков
- Максимальную длину строковых значений атрибутов

## Логирование

### Пайплайн

```
LoggerProvider -> Logger -> LogRecord -> LogProcessor -> LogExporter -> Transport
```

Архитектура зеркальна трассировке. `ОтелЗаписьЛога` содержит тело, серьезность, атрибуты, ссылку на SpanContext (для корреляции с трассировкой).

Процессоры аналогичны: простой (синхронный), пакетный (асинхронный), композитный.

### Корреляция с трассировкой

При создании записи лога автоматически захватывается текущий SpanContext из `ОтелКонтекст`. Это позволяет в бэкенде (Grafana, Jaeger) связывать логи с трассировками.

### Интеграция с logos

`ОтелАппендерLogos` реализует интерфейс аппендера библиотеки logos. Маппинг уровней:
- logos `Отладка` -> OTel `Debug`
- logos `Информация` -> OTel `Info`
- logos `Предупреждение` -> OTel `Warn`
- logos `Ошибка` -> OTel `Error`
- logos `Критичный` -> OTel `Fatal`

## Метрики

### Пайплайн

```
MeterProvider -> Meter -> Instrument -> PeriodicMetricReader -> MetricExporter -> Transport
```

### Синхронные инструменты

- `ОтелСчетчик` (Counter) - монотонно возрастающий. `Добавить(Значение, Атрибуты)`.
- `ОтелРеверсивныйСчетчик` (UpDownCounter) - может уменьшаться. `Добавить(Значение, Атрибуты)`.
- `ОтелДатчик` (Gauge) - последнее значение. `Записать(Значение, Атрибуты)`.
- `ОтелГистограмма` (Histogram) - распределение значений по бакетам. `Записать(Значение, Атрибуты)`.
- `ОтелЭкспоненциальнаяГистограмма` (ExponentialHistogram) - Base2 exponential buckets.

### Наблюдаемые (асинхронные) инструменты

- `ОтелНаблюдаемыйСчетчик` - callback вызывается при сборе
- `ОтелНаблюдаемыйРеверсивныйСчетчик`
- `ОтелНаблюдаемыйДатчик`

Callback-функция вызывается `ОтелПериодическийЧитательМетрик` при каждом цикле сбора.

### Агрегация

Каждый инструмент хранит точки данных (data points), сгруппированные по уникальным наборам атрибутов. При записи значения находится или создается точка данных с соответствующими атрибутами, и значение агрегируется:

- Sum (для Counter, UpDownCounter)
- LastValue (для Gauge)
- ExplicitBucketHistogram (для Histogram) - бакеты по умолчанию: [0, 5, 10, 25, 50, 75, 100, 250, 500, 750, 1000, 2500, 5000, 7500, 10000]
- Base2ExponentialBucketHistogram (для ExponentialHistogram)

### Представления (Views)

`ОтелПредставление` + `ОтелСелекторИнструментов` позволяют:
- Переименовать метрику
- Изменить агрегацию
- Отфильтровать атрибуты
- Установить границы бакетов гистограммы

### Временная агрегация

`ОтелВременнаяАгрегация`:
- `Кумулятивная` - значения с начала работы (по умолчанию)
- `Дельта` - значения с последнего экспорта

`ОтелСелекторВременнойАгрегации` предоставляет стратегии: `ВсегдаКумулятивная`, `ПредпочтительноДельта`, `НизкоеПотреблениеПамяти`.

### PeriodicMetricReader

`ОтелПериодическийЧитательМетрик` запускает `ФоновоеЗадание`, которое с заданным интервалом (по умолчанию 60 сек):
1. Вызывает callback'и наблюдаемых инструментов
2. Собирает точки данных со всех зарегистрированных метров
3. Передает данные экспортеру

## Экспорт

### Транспорты

Все экспортеры работают через абстракцию транспорта. Экспортер формирует OTLP-сообщение (Соответствие/JSON), передает транспорту, транспорт отправляет.

- `ОтелHttpТранспорт` - отправляет JSON по HTTP POST на эндпоинты `/v1/traces`, `/v1/logs`, `/v1/metrics`. Использует библиотеку 1connector.
- `ОтелGrpcТранспорт` - отправляет protobuf по gRPC. Использует библиотеку oint (OPI_GRPC, Rust-клиент tonic под капотом). Соединение открывается в конструкторе (eager init). Перед отправкой JSON конвертируется в protobuf формат (snake_case) через `ОтелКонвертерOtlpВProto`.

  **Ограничения схем endpoint.** OPI_GRPC принимает endpoint только в виде `http://host:port` или `https://host:port` (схема обязательна). DNS-резолвинг выполняется средствами рантайма автоматически, без отдельной схемы `dns://`. Схемы `unix://`, `ipv4:`, `ipv6:` и формат `host:port` без явной схемы не поддерживаются нижележащим клиентом - это ограничение tonic, а не нашего кода. Спецификация OTLP формулирует поддержку этих схем как SHOULD; реализация ограничена возможностями платформы.
- `ОтелВПамятьТранспорт` - сохраняет данные в массив, для тестирования.

### Экспортеры

Три экспортера (`ОтелЭкспортерСпанов`, `ОтелЭкспортерЛогов`, `ОтелЭкспортерМетрик`) преобразуют внутренние объекты SDK в формат OTLP (ExportTraceServiceRequest, ExportLogsServiceRequest, ExportMetricsServiceRequest) и передают транспорту.

### Формат OTLP

SDK формирует данные в формате OTLP JSON (camelCase). Для gRPC-транспорта `ОтелКонвертерOtlpВProto` рекурсивно преобразует ключи из camelCase в snake_case, т.к. protobuf использует snake_case имена полей.

## Пропагация контекста

### W3C TraceContext

`ОтелW3CПропагатор` реализует [W3C Trace Context](https://www.w3.org/TR/trace-context/):
- `Внедрить(Контекст, Заголовки)` - извлекает SpanContext из контекста, записывает `traceparent` и `tracestate` в Соответствие заголовков
- `Извлечь(Контекст, Заголовки)` - читает `traceparent`, возвращает новый контекст с удаленным `ОтелКонтекстСпана`

Формат traceparent: `{version}-{trace-id}-{parent-id}-{trace-flags}` (например `00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`)

### W3C Baggage

`ОтелW3CBaggageПропагатор` реализует [W3C Baggage](https://www.w3.org/TR/baggage/):
- `Внедрить(Контекст, Заголовки)` - извлекает Baggage из контекста, записывает заголовок `baggage` в формате `key1=value1,key2=value2`
- `Извлечь(Контекст, Заголовки)` - читает заголовок `baggage`, возвращает новый контекст с `ОтелBaggage`

### Композитный пропагатор

`ОтелКомпозитныйПропагатор` объединяет несколько пропагаторов. По умолчанию автоконфигурация создает композитный пропагатор из TraceContext + Baggage. Доступ к пропагаторам: `ОтелГлобальный.ПолучитьПропагаторы()`.

Если глобальный пропагатор не был установлен явно, `ОтелГлобальный.ПолучитьПропагаторы()` lazy-создаёт композитный (W3C TraceContext + W3C Baggage) через `АтомарнаяСсылка` с CAS-инициализацией - этого требует спека (Propagators API SHOULD provide a default). Возврат No-op запрещён, если SDK скомпонован.

### B3 пропагатор (отдельный пакет)

B3 (Zipkin) пропагатор распространяется **отдельным пакетом** [`opentelemetry-propagator-b3`](https://github.com/nixel2007/opentelemetry-propagator-b3) - этого требует [спецификация](https://opentelemetry.io/docs/specs/otel/context/api-propagators/#b3-requirements) («B3 MUST be distributed as a plugin/extension package»). Основной SDK ссылок на этот пакет **не имеет**.

Автоконфигурация в `ОтелАвтоконфигурация.СоздатьПропагаторы()` распознаёт значения `b3` и `b3multi` в `OTEL_PROPAGATORS` и подгружает класс через рефлексивный конструктор `Новый("ОтелB3Пропагатор", ...)`. Если пакет не установлен, выводится warning через logos и пропагатор пропускается. Это позволяет основному SDK не зависеть от B3, но прозрачно использовать его при наличии.

## Автоконфигурация

`ОтелАвтоконфигурация` - модуль, реализующий паттерн автоконфигурации из Java SDK. Читает параметры из менеджера параметров (configor), который поддерживает:
- Переменные окружения (`OTEL_SERVICE_NAME` -> `otel.service.name`)
- Файлы конфигурации (JSON, YAML)
- Программное создание

### Что создает автоконфигурация

`Инициализировать()` создает полностью настроенный SDK:
1. **Ресурс** с `service.name`
2. **Транспорт** (HTTP или gRPC в зависимости от `otel.exporter.otlp.protocol`)
3. **Провайдер трассировки** с пакетным процессором и семплером
4. **Провайдер логирования** с пакетным процессором
5. **Провайдер метрик** с периодическим читателем
6. **Пропагаторы** (TraceContext + Baggage по умолчанию)

Транспорт создается для каждого сигнала отдельно через `СоздатьТранспортДляСигнала`, что позволяет использовать разные эндпоинты, протоколы и заголовки для traces, logs и metrics.

### NoOp-режим

При `otel.enabled = false` создается SDK без экспортеров, с семплером `ВсегдаВыключен`. Все вызовы API работают, но данные никуда не отправляются. Это позволяет не менять код при отключении телеметрии.

## Многопоточность

### Модель потоков OneScript

OneScript поддерживает `ФоновыеЗадания` - реальные потоки, разделяющие глобальное состояние. Переменные модулей разделяются между потоками.

### Thread-safety в SDK

- **ОтелКонтекст** - per-thread стек, использует `ФоновыеЗадания.ПолучитьТекущее().УникальныйИдентификатор` как ключ потока. Каждый поток имеет свой стек спанов. Доступ защищен `БлокировкаРесурса`.
- **ОтелАтрибуты** - не thread-safe. Атрибуты спана устанавливаются из одного потока (создатель спана).
- **Пакетный процессор** - использует `БлокировкаРесурса` для потокобезопасного доступа к буферу спанов. Несколько потоков могут завершать спаны одновременно.
- **Метрики** - точки данных защищены `БлокировкаРесурса`. Несколько потоков могут записывать в один счетчик.
- **gRPC-транспорт** - соединение открывается в конструкторе (до запуска фоновых потоков), предотвращая race condition при инициализации нативной компоненты OPI_GRPC.

### Жизненный цикл фоновых задач

1. При автоконфигурации создаются процессоры и читатели
2. `ПакетныйПроцессор.ЗапуститьФоновыйЭкспорт()` запускает ФоновоеЗадание для экспорта спанов/логов
3. `ПериодическийЧитательМетрик.Запустить()` запускает ФоновоеЗадание для сбора и экспорта метрик
4. При `Закрыть()` фоновые задания останавливаются, оставшиеся данные экспортируются (flush)
